# RELAY Dispatch — v1.3 Roadmap

Priorities top-to-bottom. Each feature lists the service decision (where one is needed),
the architecture, and an execution checklist. Cross-cutting rules at the bottom apply to everything.

---

## 1. Maps — routing & dispatch (top priority)

**Status: ✅ Complete (2026-07-22)** — relay-route function, Schedule route panel, Today's Schedule drive chips, per-user start location, navigate deep-links, and Deputy routing (ROUTE_PLAN / DRIVE_TIME) all shipped. Maps flag is on for all users.

**Goal:** Office staff can see estimated travel times across a technician's day —
`start location → job 1 → job 2 → … → back to start` — and use it to pick the best
job order. Deputy can answer routing questions.

**Foundation already built (v1.2):** geocode-once strategy (`geo` jsonb on jobs/customers,
`src/utils/geocode.js` cache, `supabase/functions/relay-geocode` proxy, two-key Google setup:
secret server key + restricted `VITE_GOOGLE_MAPS_BROWSER_KEY`).

**Service:** Google **Routes API** (`computeRoutes`) — already on Google billing.
- `optimizeWaypointOrder: true` gives the best job order in one call.
- Called **server-side only** (Supabase Edge Function using the secret key, same pattern as relay-geocode).
- Traffic-aware ETAs via `TRAFFIC_AWARE` routing preference.

**Start location:**
- Company office = default (from Settings → company address; geocode once, store in company settings `geo`).
- Per-user override: new profile setting `startLocation` (address + geo). Falls back to company office when unset.

**Integration checklist:**
- [ ] `relay-route` server function: input = ordered/unordered job ids + start geo; output = per-leg ETAs, total drive time, optimized order.
- [ ] Schedule page: "Route view" for a selected tech + day — leg times between allocations, total, "optimize order" button that re-sequences allocations.
- [ ] **Today's Schedule widget**: show drive time chips between consecutive jobs + total for the day.
- [ ] Per-user start location in Profile settings (default = company office).
- [ ] Deputy: "what's the best order for Grace's jobs tomorrow?", "how long is the drive from job A to job B?" → call the same route function.
- [ ] Free wins: "Navigate" deep-links (`https://www.google.com/maps/dir/?api=1...`) on job detail + schedule blocks (zero API cost).

## 3. Online payments

**Goal:** Customers pay invoices online — from the portal, from emailed/SMS'd links, from the invoice PDF.

**Service: Stripe** (recommended — no real competition for this use case in AU).
- Invoice PDFs already have a mocked "Pay Invoice Online" button and a `paymentStripe` toggle in documentTheme — wire them up for real.
- Use **Stripe Checkout Sessions** (or Payment Links for zero-backend simplicity) created by a Supabase Edge Function; store `stripeSessionId` on the invoice.
- **Webhook** (Supabase Edge Function) on `checkout.session.completed` → mark invoice **Paid** (sets `paidDate`, which also feeds the job-status flow correctly since jobs are already marked Invoiced on Send).
- Keys: secret key server-side only (Supabase secrets). Publishable key not needed if using hosted Checkout.
- Surcharging/GST: include GST line from invoice totals as-is; keep Stripe fee absorption a Settings toggle for later.

**Integration checklist:**
- [ ] `create-payment-link` + `stripe-webhook` Edge Functions (a started pair already exists in the legacy `netlify/functions/` dir — port, don't rewrite).
- [ ] "Pay online" button live in: customer portal invoice view, invoice PDF/preview, email template (#5), SMS template (#6).
- [ ] Settings → Payments: connect status, currency (AUD), enable/disable per-document-type.
- [ ] Invoice detail: show payment status/link, "copy payment link".

## 4. Weather

**Service:** **Google Weather API** (aligns with existing Google billing — user preference).
*Fallback/alternative:* Open-Meteo is free + keyless (AU coverage is good) if Google's
Weather API preview access or pricing disappoints. Severe-weather **warnings** for AU are
best from the BOM public feeds regardless of forecast provider.

**Architecture:** server function `relay-weather` (same proxy pattern), cached per
lat/lng+day in the store (weather for a town changes hourly, not per-request — cache
aggressively, e.g. 1h TTL, to stay in free tiers).

**Integration checklist:**
- [ ] `relay-weather` function + client cache.
- [ ] **Weather Forecast widget**: real forecast for company office location (or user start location).
- [ ] **Today's Schedule widget**: small weather icon + temp per job (job site geo), warning badge when severe.
- [ ] Schedule page day headers: forecast glyphs.
- [ ] Deputy: "will it rain at the Newcastle job tomorrow?", proactive warnings ("3 of today's jobs are under a severe wind warning").
- [ ] Job Tracking hook (#7): snapshot weather onto the job record on its scheduled day.

## 5. Email + domain

**Service: Resend** (recommended) — modern API, domain verification via DNS records surfaced
in-app, generous free tier, React-free HTML templates fine. *Alternatives:* Postmark
(best-in-class deliverability, pricier), SendGrid (legacy-feeling).

**Architecture:** all sending via a `relay-email` Supabase Edge Function (API key server-side).
Per-company sender config stored in company settings.

**Integration checklist:**
- [ ] Settings → **Email & Domain page** (dedicated): custom domain setup wizard (shows the DNS records to add + verify button), from-name/from-address, reply-to, signature, per-template enable.
- [ ] Templates: quote (with accept link), invoice (with pay link from #3), payment receipt, appointment reminder, portal invite. Themed with documentTheme colours.
- [ ] Send actions: Quote detail "Email quote", Invoice detail "Email invoice", automatic payment reminders (X days before/after due — Settings-configurable), portal invites.
- [ ] Log sends into an `email_log` (or activity) so Deputy/Reports can see them.

**Goal:** capture actuals vs estimates + context (weather, travel, traffic), so Reports AI
and Deputy can explain wins/losses; roll data up into customer & contractor profiles.

**Capture (per job):**

- [ ] Reports: "estimate accuracy" report (jobs over/under estimate, by type/tech/customer), travel-time overhead report.

**Profiles:**

## Cross-cutting rules (apply to every feature above)

1. **Cloud-gated:** everything that costs money/API calls is gated behind `store._isCloudUser()` — local/offline accounts see the feature disabled with an upgrade hint.
2. **Secrets server-side only:** all API keys (Routes, Weather, Stripe, Resend, Twilio) live in Supabase Edge Function secrets — never in the client bundle. Only the restricted `VITE_GOOGLE_MAPS_BROWSER_KEY` ships to the client. (Netlify is retired — `netlify.toml` + `netlify/functions/` are legacy leftovers pending cleanup.)
3. **Cache first:** geocode-once pattern extends to routes (cache per job-set+day) and weather (1h TTL) to stay inside free tiers.
4. **Schema discipline:** every new table/column ships as a `supabase/migrations/00X_*.sql` file **plus** the matching `TABLE_MAP`/`TABLE_COLUMNS`/normalize/denormalize entries in `store.js` in the same commit (we've been burned by drift twice).
5. **Deputy integration** is a first-class deliverable of #1, #4, #7 — each server function should be callable both from UI and from Deputy's command handler.

## Release strategy

- **v1.3.0 ships once, complete.** All seven features above are built and tested before the
  version bumps — no per-feature point releases. Work through them top-to-bottom, one at a time.
- **1.2.x point releases** are reserved for bug fixes, quality-of-life tweaks, and customer asks
  while 1.3 is in development.
- **Guard rail:** because 1.2.x patches ship from the same `main` branch, any unfinished 1.3
  feature must be **hidden/flagged off** (no UI entry point) until it's done — otherwise a bugfix
  release would leak half-built features to the desktop clients. Build each feature dark, flip it
  on when 1.3.0 cuts.
- **Delivery:** Electron is the sole channel for now; browser hosting returns at full launch
  (architecture unaffected — Edge Functions serve both).
